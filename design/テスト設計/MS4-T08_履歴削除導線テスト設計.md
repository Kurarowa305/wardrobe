# MS4-T08 履歴削除導線テスト設計

## 目的
- MS4-T08（履歴削除導線）の完了条件を継続的に検証する。
- 履歴詳細画面から削除操作を実行し、遷移元文脈に応じて戻ること、および削除失敗時の通知を CI で担保する。

## 対象
- `apps/web/src/components/app/screens/HistoryDetailScreen.tsx`
- `apps/web/src/features/history/strings.ts`
- `apps/web/scripts/check-history-delete-flow-spec.mjs`
- `.github/workflows/ci.yml`

## テスト観点

### HDL-01 履歴詳細画面が delete Mutation を利用する
- 観点: 詳細画面から物理削除 API を呼び出せること
- 期待結果:
  - `useDeleteHistoryMutation(wardrobeId, historyId)` を利用する。
  - 削除確定後に `deleteMutation.mutateAsync()` を呼び出す。

### HDL-02 削除確認に共通文言を利用する
- 観点: 服/テンプレートと同じ削除確認体験を保てること
- 期待結果:
  - `COMMON_STRINGS.dialogs.confirmDelete.title` を参照する。
  - `COMMON_STRINGS.dialogs.confirmDelete.message` を参照する。
  - `window.confirm(...)` でユーザー確認を行う。

### HDL-03 削除成功時に from 文脈に応じて戻る
- 観点: 画面仕様の「ホームから来た場合はホーム、それ以外は履歴一覧」を維持すること
- 期待結果:
  - `useSearchParams()` で `from` を読む。
  - `resolveHistoryDetailBackHref(...)` で戻り先を解決する。
  - 削除成功後に `router.push(backHref)` を実行する。

### HDL-04 削除失敗時に destructive toast を表示する
- 観点: 削除失敗がユーザーに即時に伝わること
- 期待結果:
  - `useToast()` を利用する。
  - `HISTORY_STRINGS.detail.messages.deleteError` を destructive toast の title に使う。
  - `history/strings.ts` に削除失敗文言が定義される。

### HDL-05 ヘッダーメニューの削除項目が削除中に無効化される
- 観点: 二重送信を防ぎ、画面側の削除導線を固定化できること
- 期待結果:
  - `headerActions` に `key: "delete"` の項目がある。
  - `onSelect: handleDelete` を持つ。
  - `deleteMutation.isPending` を用いて `disabled` を制御する。

## テストスクリプト
- `pnpm --filter web test:history-delete-flow`
  - `apps/web/scripts/check-history-delete-flow-spec.mjs` を実行し、上記 HDL-01〜HDL-05 を静的検証する。

## CI 組み込み
- `.github/workflows/ci.yml` に `History delete flow spec test` ステップを追加し、push 時に自動実行する。

## PRサマリーに記載するテストケース
- HDL-01: 履歴詳細画面が `useDeleteHistoryMutation` と `mutateAsync()` で削除処理を実行する。
- HDL-02: 削除確認ダイアログが共通文言 `削除しますか？ / この操作は取り消せません` を使う。
- HDL-03: 削除成功時に `from` 文脈から解決した戻り先へ遷移する。
- HDL-04: 削除失敗時に destructive toast でエラー通知する。
- HDL-05: ヘッダーメニューの削除項目が onSelect と削除中無効化に対応する。
