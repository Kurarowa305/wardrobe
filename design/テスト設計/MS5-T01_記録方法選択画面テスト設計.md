# MS5-T01 記録方法選択画面テスト設計

## 目的

- MS5-T01（記録：方法選択画面）の完了条件を継続的に検証する
- ホームから遷移したユーザーが、2つの記録方法を迷わず選べるUI構成をCIで担保する

## 対象

- `apps/web/src/components/app/screens/RecordMethodScreen.tsx`
- `apps/web/src/features/record/strings.ts`
- `apps/web/scripts/check-record-method-screen-spec.mjs`
- `.github/workflows/ci.yml`

## テスト観点

### RMS-01 記録方法選択画面が所定ファイルに存在する
- 観点: 記録フローの起点画面が画面設計どおりの配置に存在すること

### RMS-02 AppLayout でタイトルとホーム戻り導線を定義する
- 観点: 記録画面がスタック画面としてヘッダーと戻る導線を備えること
- 確認項目:
  - `title={RECORD_STRINGS.method.title}` を指定する
  - `backHref={ROUTES.home(wardrobeId)}` を指定する

### RMS-03 方法選択メッセージと2つの選択肢を表示する
- 観点: ユーザーが何をすべき画面か理解でき、選択肢が過不足なく提示されること
- 確認項目:
  - `RECORD_STRINGS.method.message`
  - `RECORD_STRINGS.method.actions.byTemplate`
  - `RECORD_STRINGS.method.actions.byCombination`

### RMS-04 2つの選択肢がそれぞれ専用の記録画面へ遷移する
- 観点: 方法選択後にテンプレート記録または服組み合わせ記録へ正しく分岐できること
- 確認項目:
  - `ROUTES.recordByTemplate(wardrobeId)` を利用する
  - `ROUTES.recordByCombination(wardrobeId)` を利用する

### RMS-05 選択肢が Card 内の outline ボタンとして並ぶ
- 観点: UIコンポーネント設計の BasicCard（2択）に沿った視認性の高いレイアウトであること
- 確認項目:
  - `Card` / `CardContent` を利用する
  - `variant="outline"` のボタンを利用する
  - 横幅いっぱい・左寄せのボタンスタイルを指定する

### RMS-06 方法選択画面向け文言が record strings に定義される
- 観点: 画面固有文言が集中管理され、後続画面と同じ方針で保守できること
- 確認項目:
  - `title`
  - `message`
  - `byTemplate`
  - `byCombination`

## テスト実装

- スクリプト: `pnpm --filter web test:record-method-screen`
- 実体: `apps/web/scripts/check-record-method-screen-spec.mjs`
- CI適用:
  - `.github/workflows/ci.yml` に `Record method screen spec test` を追加し、PR時に自動検証する

## PRサマリーに記載するテストケース

- RMS-01 記録方法選択画面のファイル配置
- RMS-02 AppLayout のタイトル・ホーム戻り導線
- RMS-03 方法選択メッセージ + 2択表示
- RMS-04 テンプレート記録 / 組み合わせ記録への遷移
- RMS-05 Card + outline ボタンのレイアウト
- RMS-06 方法選択画面向け文言定義
