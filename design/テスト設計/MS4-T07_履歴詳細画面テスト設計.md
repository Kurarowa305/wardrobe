# MS4-T07 履歴詳細画面テスト設計

## 目的
- 履歴詳細画面が History API の詳細取得結果を用いて、表示専用のUIを設計どおりに構成していることを静的検証する。
- 戻り先文脈付きルーティングを維持したまま、履歴詳細ページが必要な静的パスを生成できることを確認する。

## 対象
- `apps/web/src/components/app/screens/HistoryDetailScreen.tsx`
- `apps/web/src/app/wardrobes/[wardrobeId]/(stack)/histories/[historyId]/page.tsx`
- `apps/web/src/features/history/strings.ts`
- `apps/web/scripts/check-history-detail-screen-spec.mjs`

## テスト観点
1. `useHistory(wardrobeId, historyId)` を利用した詳細取得と、ロード中・エラー時の表示分岐を持つ。
2. 履歴詳細が日付、入力方法、組み合わせ/テンプレート文脈、着用服一覧を表示する。
3. 着用服一覧は服詳細へのリンクを持たず、画像URL解決・no image・削除済み表示に対応する。
4. ヘッダーメニューに編集/削除項目が存在する。
5. 履歴詳細ページが `historyId` を `HistoryDetailScreen` に渡し、fixture由来IDと `hs_mock_` 系IDの静的パスを生成する。
6. 履歴詳細向け文言が `history/strings.ts` に定義される。

## テストケース
- **HDS-01**: 履歴詳細画面ファイルが存在する。
- **HDS-02**: `useHistory` を利用し、`isPending` / `isError` / エラー表示分岐を備える。
- **HDS-03**: 日付・入力方法・着用服一覧・着用回数・最終着用日を表示する。
- **HDS-04**: 着用服一覧が非リンク表示で、画像・プレースホルダ・削除済み表示に対応する。
- **HDS-05**: ヘッダーメニューに編集/削除がある。
- **HDS-06**: 詳細ページが fixture と mock ID の静的パスを生成し、`historyId` を screen に渡す。
- **HDS-07**: 履歴詳細向け文言が strings に定義される。

## 実行コマンド
- `pnpm --filter web test:history-detail-screen`
