# ワードローブ作成バリデーションテスト設計

## 1. 目的

- ワードローブ作成画面で、ワードローブ名未入力時に作成ボタンが非活性になることを継続的に検証する
- 入力エラーのトースト通知を廃止し、画面操作で未入力送信を防ぐ方針をCIで担保する
- 作成成功時のホーム遷移導線が従来どおり維持されることを静的テストで保証する

## 2. 対象

- `apps/web/src/components/app/screens/WardrobeCreateScreen.tsx`
- `apps/web/src/features/wardrobe/strings.ts`
- `apps/web/scripts/check-wardrobe-create-validation-spec.mjs`
- `.github/workflows/ci.yml`

## 3. テストケース

### WCV-01 入力値を trim した結果で送信可否を判定する
- 観点: 空白のみ入力を未入力として扱えること
- 期待結果:
  - `WardrobeCreateScreen.tsx` に `const trimmedName = name.trim();` がある
  - `trimmedName.length === 0` を用いた非活性判定がある

### WCV-02 ワードローブ名未入力時は作成ボタンが非活性になる
- 観点: 未入力送信をUI上で防止できること
- 期待結果: `Button` に `disabled={isSubmitDisabled}` が指定されている

### WCV-03 未入力時のトースト導線を削除する
- 観点: 入力エラー通知をトーストではなくボタン非活性に統一できていること
- 期待結果:
  - `WardrobeCreateScreen.tsx` に `useToast` の import が存在しない
  - `toast(` と `variant: "destructive"` が存在しない
  - `WARDROBE_STRINGS.create.errors` 参照が存在しない

### WCV-04 文言定義から未使用の入力エラー文言を削除する
- 観点: 文言定義と画面実装の乖離防止
- 期待結果: `src/features/wardrobe/strings.ts` に `errors` / `nameRequired` が存在しない

### WCV-05 入力済みの場合のみAPI作成結果のwardrobeIdでホームへ遷移する
- 観点: 正常系の遷移が固定IDではなくAPIレスポンスに追従すること
- 期待結果:
  - `createWardrobeMutation.mutateAsync({ name: trimmedName })` が呼ばれる
  - `ROUTES.home(created.wardrobeId)` で遷移する
  - `DEMO_IDS.wardrobe` を遷移に利用しない

### WCV-06 API呼び出し中は多重送信を防止する
- 観点: ダブルタップなどによる重複作成を防げること
- 期待結果:
  - `if (isSubmitDisabled || createWardrobeMutation.isPending)` で早期returnする
  - `Button` の `disabled` に `createWardrobeMutation.isPending` が含まれる

## 4. 実行方法

- 実行コマンド: `pnpm --filter web test:wardrobe-create-validation`
- 実装方式: Node.js による静的検査スクリプト

## 5. CI組み込み

- `.github/workflows/ci.yml` に `Wardrobe create validation spec test` を追加する
- push 時に自動実行し、バリデーション仕様の退行を検知する
