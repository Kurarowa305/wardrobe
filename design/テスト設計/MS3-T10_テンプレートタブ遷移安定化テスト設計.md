# MS3-T10 テンプレートタブ遷移安定化テスト設計

## 目的

- テンプレートタブ遷移時のみフルリロードが発生する退行を防ぐ
- `templates` 画面ルートと `templates` APIモックのパス衝突時に、MSWがNext.jsルーティング通信を横取りしないことをCIで担保する

## 背景

- 画面ルート: `/wardrobes/{wardrobeId}/templates`
- Template APIモック: `GET /wardrobes/{wardrobeId}/templates`
- 上記が同一パスのため、MSWハンドラにガードがない場合、Next.js のRSC/Prefetch/Navigation通信をAPIレスポンスで上書きし、クライアント遷移失敗の起点になる

## 対象スクリプト

- `apps/web/scripts/check-template-tab-navigation-spec.mjs`
- 実行コマンド: `pnpm --filter web test:template-tab-navigation`

## テストケース

### TTN-01 Template MSW handler が存在する
- 観点: 対象実装ファイルの配置保証
- 期待結果: `apps/web/src/mocks/handlers/template.ts` が存在する

### TTN-02 Template MSW handler が passthrough を import する
- 観点: APIモックのバイパス処理を使える状態の保証
- 期待結果: `template.ts` が `passthrough` を import する

### TTN-03 Next.js ルーティング系リクエスト判定ガードを持つ
- 観点: 画面遷移通信をAPIモック対象外にする条件定義
- 期待結果:
  - `shouldBypassTemplateApiMock(request)` を定義する
  - `navigate/document` 判定を含む
  - `rsc` / `next-router-prefetch` / `next-router-state-tree` ヘッダ判定を含む
  - `text/x-component` / `text/html` の Accept 判定を含む

### TTN-04 Template handlers 全てでガード時に passthrough する
- 観点: 一部ハンドラのみ未対応の取りこぼし防止
- 期待結果: template の5ハンドラ（一覧/詳細/作成/更新/削除）すべてがガード時に `passthrough()` を返す

### TTN-05 ガード判定がシナリオ適用より先に実行される
- 観点: `applyMockScenario` がルーティング通信に影響しない順序保証
- 期待結果: 各ハンドラで `shouldBypassTemplateApiMock` 判定が `applyMockScenario` より前に配置される

## CI適用

- `.github/workflows/ci.yml` に `Template tab navigation spec test` を追加し、PR時に自動検証する
