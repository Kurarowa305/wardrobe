# ルーティングテスト設計

## 1. 目的

* MS0-T02 のルーティング骨格（`wardrobeId` 配下ルーティング、タブ＋スタック、戻る動線）を継続的に検証する
* 仕様逸脱を CI で早期検知する

---

## 2. 対象

* `apps/web/src/app` のルーティング定義
* `apps/web/src/components/app/layout` / `navigation` / `screens` のルーティング関連実装
* `apps/web/src/features/history/routing.ts`（履歴詳細の `from` クエリ戻り先解決）
* `apps/web/src/constants/routes.ts`

---

## 3. 実行方法

* ローカル: `pnpm --filter web test:routing`
* CI: `.github/workflows/ci.yml` の `Routing spec test` ステップで実行
* 実装スクリプト: `apps/web/scripts/check-routing-spec.mjs`

---

## 4. テスト項目一覧

| ID | 観点 | 検証内容 |
| --- | --- | --- |
| RT-01 | ルート初期遷移 | `/` が `/wardrobes/new` にリダイレクトされる |
| RT-02 | 作成画面URL | ワードローブ作成画面が `/wardrobes/new` に存在する |
| RT-03 | 旧URL遮断 | 旧トップレベルルート (`/home` 等) の実体が残っていない |
| RT-04 | スタック到達 | `wardrobeId` 配下のスタック画面群が存在する |
| RT-05 | 画面総数 | 作成1 + `wardrobeId`配下14 = 合計15画面が定義されている |
| RT-06 | タブ表示制御 | タブ画面のみ TabBar が表示条件を満たす |
| RT-07 | タブ遷移ID維持 | TabBar の全リンクが `wardrobeId` を保持する |
| RT-08 | タブ戻る非表示 | タブ4画面で `backHref` を持たない |
| RT-09 | スタック戻る表示 | スタック画面（履歴詳細含む）で戻る導線を持つ |
| RT-10 | 記録戻り先 | 記録方法選択の戻り先がホーム（JSX / createElement どちらの記法でも可） |
| RT-11 | 記録下位戻り先 | 記録下位2画面の戻り先が記録方法選択（JSX / createElement どちらの記法でも可） |
| RT-12 | テンプレ戻り先 | 追加/詳細/編集の戻り先が仕様通り |
| RT-13 | 服戻り先 | 追加/詳細/編集の戻り先が仕様通り |
| RT-14 | 履歴詳細戻り先(home) | `from=home` でホームへ戻る |
| RT-15 | 履歴詳細戻り先(histories) | `from=histories` で履歴一覧へ戻る |
| RT-16 | 履歴詳細戻り先(異常値) | `from` が不正値の時は履歴一覧へフォールバック |
| RT-17 | 履歴詳細戻り先(未指定) | `from` 未指定時は履歴一覧へフォールバック |
| RT-18 | 履歴タブ導線 | 履歴タブに「ホームタブへ」導線を置かない |
| RT-19 | wardrobeId必須 | `wardrobeId` 配下画面の route 呼び出しで `wardrobeId` が先頭引数 |
| RT-20 | 作成後導線 | 作成画面が `POST /wardrobes` の返却 `wardrobeId` のホームへ遷移する |
| RT-21 | 履歴詳細の静的生成 | `useSearchParams()` を Suspense 配下のサブツリーに隔離し、build 時の CSR bailout を防ぐ |
| RT-22 | テンプレ詳細/編集の動的ルート | fixture由来の `generateStaticParams` を持たず、runtime動的ルートとして実装される |

---

## 5. 運用ルール

* ルーティング仕様を変更した場合は、本資料と `check-routing-spec.mjs` を同時更新する
* `app/**/page.tsx` を薄いルーター運用にしたため、遷移ロジック変更時は `components/app/screens/*Screen.tsx` も検証対象に含める
* 静的生成対象ページで `useSearchParams()` を使う場合は、最小単位の Suspense 境界で囲み、fallback の戻り先も仕様上安全な値にする
* CI で失敗した場合は、仕様変更か実装逸脱かを先に判定してから修正する
