# 服ジャンル追加テスト設計

## 目的
- 服に `genre`（トップス / ボトムス / その他）が追加された後も、一覧・服作成/編集・テンプレート作成/編集・服の組み合わせ記録画面で UI と API が整合していることをCIで継続検証する。
- ジャンル別セクション、折りたたみ、ジャンル別「さらに読み込む」、ジャンル選択プルダウン、fixture拡張の退行を防ぐ。

## 対象スクリプト
- `apps/web/scripts/check-clothing-types-spec.mjs`
- `apps/web/scripts/check-clothing-fixtures-spec.mjs`
- `apps/web/scripts/check-clothing-api-client-spec.mjs`
- `apps/web/scripts/check-clothing-msw-handlers-spec.mjs`
- `apps/web/scripts/check-clothing-list-screen-spec.mjs`
- `apps/web/scripts/check-clothing-create-screen-spec.mjs`
- `apps/web/scripts/check-clothing-edit-screen-spec.mjs`
- `apps/web/scripts/check-template-form-screen-spec.mjs`
- `apps/web/scripts/check-record-combination-screen-spec.mjs`

## テストケース

### CG-01 Clothing schema / VM が genre を保持する
- 観点: API DTO・画面VMの基本整合
- 期待結果:
  - `ClothingGenreDto = "tops" | "bottoms" | "others"` が定義される
  - `ClothingDto` / `ClothingListItemDto` / VM が `genre` を保持する

### CG-02 Clothing fixture が各ジャンル10件以上を持つ
- 観点: ジャンル別UIとページングの検証用データ確保
- 期待結果:
  - トップス・ボトムス・その他それぞれ10件以上の fixture を持つ
  - 画像あり / 画像なし / 削除済みデータを含む

### CG-03 Clothing API / MSW handler が genre を扱う
- 観点: 一覧絞り込み・作成・更新のAPI整合
- 期待結果:
  - 一覧 query に `genre` を渡せる
  - create / update payload に `genre` を含める
  - MSW handler が `genre` の validation / 保存 / 絞り込みを行う

### CG-04 服一覧画面がジャンル別セクションを表示する
- 観点: 一覧画面の新UI要件
- 期待結果:
  - トップス / ボトムス / その他の見出しを持つ
  - 各見出し右に対応SVGアイコンを表示する
  - 各セクションが展開 / 折りたたみできる
  - 各セクションごとに「さらに読み込む」を持つ

### CG-05 服作成 / 編集画面がジャンルプルダウンを表示する
- 観点: 入力UI要件
- 期待結果:
  - `select` 形式でトップス / ボトムス / その他を選択できる
  - プルダウン右に一覧と同じアイコンを表示する
  - 未選択時のエラーメッセージを表示する

### CG-06 テンプレート作成 / 編集画面がジャンル別セクションを表示する
- 観点: 服選択UI要件
- 期待結果:
  - ジャンル別セクションで服カードを表示する
  - 各セクションで展開 / 折りたたみできる
  - 各セクションごとに追加読み込みができる

### CG-07 服の組み合わせ記録画面がジャンル別セクションを表示する
- 観点: 記録画面の選択UI要件
- 期待結果:
  - ジャンル別セクションで複数選択UIを表示する
  - 各セクションで展開 / 折りたたみできる
  - 各セクションごとに追加読み込みができる
  - 送信時に `yyyymmdd` へ変換した日付と `clothingIds` を送る

## CI適用
- 既存の `CI` workflow で上記スクリプトを継続実行する。
- 既存ステップの更新により、PR時に服ジャンル対応の仕様退行を自動検知する。
